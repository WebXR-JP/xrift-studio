# XRift Studio UX 原則

XRift Studio の体験は、華やかな演出ではなく、制作の流れを途切れさせない「和」によって成立する。ユーザーの意図、画面の状態、次にできる操作、操作後の到達点を揃え、迷いなく一つの制作行為としてつなげる。

この文書は、機能の追加、既存画面の変更、画面文言、レビューで共通に使う設計判断の基準である。実装時は `AGENT.md` のルールと `skills/xrift-studio-ux/SKILL.md` の手順も参照する。

状態ごとの動きと機能一覧は [マイクロインタラクション Wiki](./UX_INTERACTIONS.md) を正とする。新機能は、その機能 ID と参照する `MI-xx` を明記してから実装する。

## 体験の約束

### 1. 必要な変化を、必要な時に、行動できる形で届ける

更新は設定画面の奥に置かない。アプリが必要性を検知し、現在と最新の状態を比較して示し、同じ場所で更新できるようにする。後回しにする選択も残し、処理中と完了後の状態を明らかにする。

実装の型:

1. 画面表示の妨げにならないタイミングで状態を確認する。
2. 変更が必要なときだけ、理由と差分を提示する。
3. 主操作を一つに絞り、延期または閉じる操作も安全に用意する。
4. 実行中は重複操作を防ぎ、成功後は再取得した状態に更新する。

現行実装では、`App.tsx` が起動後に CLI の現在・最新バージョンを確認し、`UpdateDialog.tsx` が差分、更新、後で行う選択をまとめて提示している。

### 2. 完了を次の開始点にする

作成、起動、公開の成功を通知だけで終わらせない。ユーザーが次に見たい対象を開いた状態、または一操作で開ける状態まで進める。作成直後に新しいワールドまたはアイテムを開く、実行後に起動 URL を表示する、公開後に公開先を開くのは同じ原則の表れである。

実装の型:

1. 成功時に、結果を識別できる ID・URL・パスを確定する。
2. 成果物を選択状態にするか、結果を開く明示的な操作を表示する。
3. 成功トーストは補助として使い、結果へ行く画面状態を必ず残す。
4. 失敗時は作業中の文脈を保持し、ログ、再試行、編集画面などの復帰手段を示す。

現行実装では、作成完了後にプロジェクト一覧を再取得して新しいプロジェクトを選択し、エディタでは実行中のローカル URL と公開済み URL をその場で開ける。

### 3. 公開前に、初期状態を作り手の表現へ置き換える

テンプレートから作ったワールドやアイテムは、そのまま公開すると他の作品と区別できない。アップロードの直前に `xrift.json` のタイトル・説明とサムネイルを確認し、テンプレートの値や画像のままなら公開を開始しない。編集画面へ案内し、保存後に未完了の項目を再確認して、すべて整ったときだけアップロードを続ける。

この確認は、公開を妨げるための警告ではない。初期ワールドのように見える失敗を公開前に防ぎ、編集した内容がそのまま公開につながる導線である。初期値を判定する条件は、CLI のテンプレート変更に追随して見直す。

### 4. 一覧を制作の記憶と入口にする

プロジェクトライブラリは単なるデータの表ではなく、制作を再開するための場所である。各カードはサムネイル、名前、タイトル、説明によって見分けられ、作成の入口は一覧の先頭に常に置く。件数、保存場所、空の状態も、今どこにいるかを静かに伝える。

実装の型:

1. 一覧の最初に新規作成の入口を固定し、空の状態でも次の操作を失わせない。
2. 各項目に視覚的な手掛かりと最小限の識別情報を載せる。
3. 読み込み中、空、失敗、更新後を区別して扱う。
4. 一覧から開く、サムネイルを整える、公開先を確認するなど、制作の再開に必要な操作を近くに置く。

現行実装では、`ProjectLibrary.tsx` が先頭の新規プロジェクトカード、件数、空状態、グリッドを管理し、`ProjectCard.tsx` が種別、サムネイル、名前を主役にしている。

### 5. 状態を隠さず、静かに変化を伝える

ユーザーは操作結果だけでなく、いま何が起きているかを知る必要がある。処理中のラベル、無効化、起動中の URL、成功・失敗の通知を一貫して使う。視覚表現は落ち着いた基調を保ち、色と動きは意味を補強するためにだけ使う。

- 白とニュートラルカラーを基調にする。ブランド色は主操作と注目すべき結果に限定する。
- 通常、ホバー、無効、実行中、成功、失敗を視覚と文言の両方で区別する。
- 短いフェードや移動は、表示されたことや画面が切り替わったことを示す範囲に留める。
- 主要操作はアイコンだけにせず、読める操作名を添える。Markdown 文書と画面文言には絵文字を使わない。

## 新機能の設計手順

1. ユーザーが達成したいことを一文で書く。内部操作ではなく、得たい状態を主語にする。
2. 操作前、処理中、成功、失敗、一覧や前画面への復帰の五つの状態を書き出す。
3. 各状態で、画面が示す事実、主操作、戻る先を決める。主操作は一つに絞る。
4. 成功時に次の目的地を開くか、目的地をすぐ開ける状態を残す。
5. 既存のライブラリ、エディタ、更新ダイアログのどこに戻るかを決め、一覧の情報も更新する。
6. 実装後に実画面で確認し、操作名だけで次の一手が分かるかを検証する。

## レビュー用チェックリスト

- [ ] 画面を開いた直後に、対象の状態と主操作が分かる。
- [ ] 作成・起動・公開・更新の成功後に、結果そのものへ到達できる。
- [ ] アップロード前にタイトル、説明、サムネイルがテンプレートのままではないことを確認し、未完了なら編集から公開へ戻れる。
- [ ] 実行中の表示、二重操作の防止、安全な中断の可否が一致している。
- [ ] 失敗時に、原因を確認する場所または再試行の方法がある。
- [ ] 一覧には新規作成の入口、項目を見分ける情報、空状態の次の一手がある。
- [ ] 色、動き、アイコンが装飾ではなく状態や操作の意味を補助している。
- [ ] 画面文言と Markdown 文書に絵文字がなく、主要操作を言葉で理解できる。

## 避けること

- 成功トーストだけを出して、成果物や URL への導線を残さない。
- 更新の必要性や差分を示さず、更新を強いる。
- 一覧を空白にし、新規作成や再開の入口を失わせる。
- 実行中なのに通常時と同じ操作を可能にし、二重実行や安全でない閉じ方を招く。
- ブランド色、強い影、動きを常用して、どれが重要な状態か分からなくする。
- アイコンや絵文字だけに操作や状態の意味を持たせる。
